## Descripción

En este proyecto, los estudiantes diseñarán y desarrollarán un sistema de software que permita a las empresas gestionar eventos corporativos. Este sistema incluirá funcionalidades como:

1.  Registro y creación de eventos.
2.  Gestión de asistentes a eventos.
3.  Panel de administración para gestionar usuarios y eventos.
4.  Funcionalidades básicas de frontend y backend con patrones de diseño (e.g., MVC o microservicios).
5.  Uso de principios de clean architecture con separación clara de responsabilidades.

## Objetivos del proyecto

*   Aplicar principios clave de la arquitectura de software (como desacoplamiento, modularidad y escalabilidad).
*   Integrar conceptos de diseño arquitectónico como patrones estructurales y anti-patrones a evitar.
*   Demostrar habilidades tanto en el desarrollo backend como frontend mientras trabajan colaborativamente en equipos equilibrados.
*   Aprender la importancia de documentar, justificar y comunicar decisiones arquitectónicas y de diseño.

## Restricciones del Proyecto

### Restricciones técnicas:

*   **Frontend:** Utilizar React o Angular (según decidan), implementación integrada con diseño intuitivo y accesibilidad.
*   **Backend:** Lenguaje obligatorio en Node.js o Python (con frameworks como Express o Flask), implementando una API RESTful completa.
*   **Base de datos:** Relacional utilizando PostgreSQL o MySQL.
*   **Infraestructura:** El proyecto debe desplegarse en un entorno de prueba utilizando contenedores Docker.
*   **Patrones arquitectónicos obligatorios:** Aplicar al menos 2 patrones arquitectónicos, como MVC, Microservicios, o CQRS, dependiendo de su diseño.
*   **Documentación obligatoria:** Documentar las decisiones arquitectónicas (ADR), incluyendo diagramas C4 (contexto, contenedores, componentes, código).
*   Crear diagrama UML de casos de uso.

### Tiempo estimado:

*   **Duración:** 12 semanas, con entregas parciales para monitorear el progreso.
*   **Entrega final:** Implementación funcional más documentación detallada en la semana 12.

### División de roles por equipo:

*   Cada equipo debe estar compuesto por:
    *   1 persona que trabaje.
    *   2 especialistas en backend.
    *   2 especialistas en frontend.
*   Total de 5 grupos con 6-7 personas por grupo (roles y responsabilidades balanceadas).

## Criterios de Evaluación (por equipo)

**Diseño arquitectónico (30%):**

*   Uso correcto de patrones de diseño.
*   Calidad y claridad en los diagramas C4.
*   Documentación adecuada de las decisiones arquitectónicas (ADR).

**Desarrollo del backend (20%):**

*   Implementación de endpoints funcionales y bien documentados.
*   Correcta modelación de base de datos y relaciones.
*   Buen manejo de modularidad, controladores y servicios.

**Desarrollo del frontend (20%):**

*   Interfaz de usuario funcional, intuitiva y accesible.
*   Correcta conexión con el backend.
*   Uso adecuado de componentes reutilizables.

**Pruebas e integración (15%):**

*   Calidad de las pruebas de integración y reporte de errores.
*   Uso de herramientas para pruebas unitarias o de backend/frontend.

**Presentación final y defensa técnica (15%):**

*   Argumentación sólida y justificada de las decisiones técnicas.
*   Demo funcional que sea clara y efectiva.

## Lineamientos adicionales para facilitar la calificación

*   **Entregas parciales:** Asegurarse que se entreguen informes claros cada dos semanas para dar retroalimentación a tiempo.
*   **Herramientas de control de versiones:** Cada equipo debe usar un repositorio Git con historial claro de commits asociados a tareas específicas.
*   **Plataforma de comunicación y gestión:** Se debe emplear una herramienta como Linear para evidenciar asignación de tareas.
